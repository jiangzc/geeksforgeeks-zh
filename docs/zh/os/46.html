操作系统中的信号量

先决条件： 进程同步 ， Mutex vs Semaphore

信号量只是一个变量。 此变量用于解决临界区问题并在多处理环境中实现进程同步。
两种最常见的信号量是计算信号量和二进制信号量。 计数信号量可以取非负整数值，二进制信号量只能取值0和1。

现在让我们看看它是如何做到的。

首先看两个可用于访问和更改信号量变量值的操作。



有关P和V操作的一些观点

  1. P操作也称为等待，休眠或降低操作，V操作也称为信号，唤醒或向上操作。
  2. 这两个操作都是原子操作，信号量总是初始化为一个。
  3. 关键部分被两个操作包围以实现过程同步。参见下面的图像。过程P的临界部分介于P和V操作之间。

现在，让我们看看它如何实现互斥。 让两个进程P1和P2和一个信号量s初始化为1.现在如果假设P1进入其临界区，则信号量s的值变为0.现在如果P2想要进入其临界区，那么它将等待直到s > 0，这只能在P1完成其临界区并在信号量s上调用V操作时发生。 这样就实现了互斥。 请查看下面的图片了解详情。

上面的描述是针对二进制信号量的，它只能取两个值0和1.还有一种称为计数信号量的信号量，它可以取大于1的值。

现在假设存在一个实例数为4的资源。现在我们初始化S = 4并且rest与二进制信号量相同。 每当进程想要该资源时它调用P或等待函数，并且当它完成时它调用V或信号函数。 如果S的值变为零，则过程必须等到S变为正。 例如，假设有4个进程P1，P2，P3，P4，它们都在S上调用等待操作（用4初始化）。 如果另一个进程P5想要资源，那么它应该等到四个进程调用中的一个发出信号函数并且信号量的值变为正数。

这个信号量实现中的问题

每当任何进程等待时，它会不断检查信号量值（在（s == 0）时查看此行;在P操作中）并浪费CPU周期。 为避免这种情况，下面提供另一种实施方 

P(Semaphore s)
{
    s = s - 1;
    if (s < 0) {

        // add process to queue
        block();
    }
}

V(Semaphore s)
{
    s = s + 1;
    if (s >= 0) {

        // remove process p from queue
        wakeup(p);
    }
}

在此实现中，只要进程等待，它就会被添加到与该信号量相关联的进程的等待队列中。 这是通过该进程上的系统调用块（）完成的。 当一个进程完成时，它调用signal函数并恢复队列中的一个进程。 它使用wakeup（）系统调用。
