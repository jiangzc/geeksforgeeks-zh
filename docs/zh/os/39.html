多处理器调度

在多处理器调度中，可以使用多个CPU ，因此可以实现负载分配 。 然而，与单处理器调度相比，多处理器调度更复杂 。 在多处理器调度中，存在处理器相同的情况，即HOMOGENEOUS，就其功能而言，我们可以使用任何可用的处理器来运行队列中的任何进程。

多处理器调度的方法 -

一种方法是所有调度决策和I / O处理由称为主服务器的单个处理器处理，而其他处理器仅执行用户代码 。 这很简单，减少了数据共享的需要。 整个场景称为非对称多处理 。

第二种方法使用对称多处理 ，其中每个处理器都是自调度的 。 所有进程可以在公共就绪队列中，或者每个处理器可以具有其自己的专用队列以用于就绪进程。 通过使每个处理器的调度程序检查就绪队列并选择要执行的进程，进一步进行调度。

处理器亲和力 -

处理器关联性意味着进程与当前正在运行的处理器具有亲缘关系 。
当进程在特定处理器上运行时，对高速缓存存储器有某些影响。 该进程最近访问的数据填充处理器的高速缓存，因此通常在高速缓冲存储器中满足该进程的连续存储器访问。 现在，如果进程迁移到另一个处理器，则必须使第一个处理器的高速缓冲存储器的内容无效，并且必须重新填充第二个处理器的高速缓存。 由于高速缓存和重新填充高速缓存的成本很高，大多数SMP（对称多处理）系统都试图避免将进程从一个处理器迁移到另一个处理器，并尝试使进程在同一处理器上运行。 这被称为PROCESSOR AFFINITY 。

有两种类型的处理器关联：



  1. 软亲和性 -当操作系统具有尝试使进程在同一处理器上运行但不保证将执行此操作的策略时，这种情况称为软关联。
  2. 硬亲和性 -某些系统（如Linux）也提供一些支持Hard Affinity的系统调用，允许进程在处理器之间进行迁移。

负载均衡 -

负载平衡是使工作负载均匀分布在SMP系统中的所有处理器上的现象 。 只有在每个处理器都有自己的私有进程队列且有资格执行的系统上，才需要进行负载均衡。 负载平衡是不必要的，因为一旦处理器变为空闲，它立即从公共运行队列中提取可运行的进程。 在SMP（对称多处理）上，重要的是保持所有处理器之间的工作负载平衡以充分利用具有多个处理器的好处，否则一个或多个处理器将处于空闲状态，而其他处理器具有高工作负载以及等待处理器的处理器列表。中央处理器。

负载平衡有两种常规方法：

  1. 推送迁移 -在推送迁移中，任务会定期检查每个处理器上的负载，如果发现不平衡，则通过将进程从过载转移到空闲或不太繁忙的处理器，均匀分配每个处理器上的负载。
  2. 拉迁移 -当空闲处理器从繁忙的处理器中拉出等待任务以执行时，会发生拉迁移。

多核处理器 -

在多核处理器中， 多个处理器核心位于同一物理芯片上。 每个核心都有一个寄存器集，用于维护其体系结构状态，因此在操作系统中显示为一个单独的物理处理器。 使用多核处理器的SMP系统比每个处理器都有自己的物理芯片的系统更快，耗电更少 。

然而，多核处理器可能使调度问题复杂化 。 当处理器访问内存时，它会花费大量时间等待数据变得可用。 这种情况称为MEMORY STALL 。 它由于各种原因而发生，例如高速缓存未命中，其正在访问不在高速缓冲存储器中的数据。 在这种情况下，处理器可以花费高达50％的时间等待数据从存储器中获得。 为了解决这个问题，最近的硬件设计已经实现了多线程处理器内核，其中为每个内核分配了两个或更多个硬件线程。 因此，如果一个线程在等待内存时停止，则内核可以切换到另一个线程。

多线程处理器有两种方法：

  1. 粗粒度多线程 -在粗粒度多线程中，线程在处理器上执行，直到发生诸如内存停顿之类的长延迟事件，由于长延迟事件引起的延迟，处理器必须切换到另一个线程以开始执行。 在线程之间切换的成本很高，因为必须在另一个线程开始在处理器核心上执行之前终止指令流水线。 一旦这个新线程开始执行，它就开始用它的指令填充管道。
  2. 细粒度多线程 -这种多线程主要在指令周期的边界处以更精细的级别在线程之间切换。 细粒度系统的架构设计包括用于线程切换的逻辑，因此线程之间切换的成本很小。

虚拟化和线程 -

在这种类型的多处理器调度中，即使单个CPU系统也像多处理器系统一样工作。 在具有虚拟化的系统中，虚拟化向系统上运行的每个虚拟机提供一个或多个虚拟CPU，然后在虚拟机之间调度物理CPU的使用。 大多数虚拟化环境都有一个主机操作系统和许多客户操作系统。 主机操作系统创建和管理虚拟机，每个虚拟机都安装了客户操作系统，并在该客户机内运行应用程序。可以为特定用例，应用程序和用户分配每个客户操作系统，包括分时或甚至真实 - 时间操作。 任何在给定时间内承担一定进展的客户操作系统调度算法都会受到虚拟化的负面影响。 在时间共享操作系统中，尝试为每个时间片分配100毫秒，以便为用户提供合理的响应时间。 给定的100毫秒时间片可能需要超过100毫秒的虚拟CPU时间。 根据系统的繁忙程度，时间片可能需要一秒或更长时间，这会导致登录到该虚拟机的用户的响应时间非常短。 这种调度分层的净效果是，各个虚拟化操作系统只接收一部分可用的CPU周期，即使他们认为它们正在接收所有周期，并且他们正在调度所有这些周期。通常，每天的时间虚拟机中的时钟不正确，因为定时器不再触发，而不是专用CPU上的时钟。

因此， 虚拟化可以撤消虚拟机内操作系统的良好调度算法工作。
