虚拟内存

虚拟内存是一种存储分配方案，其中辅助内存可以被寻址，就像它是主内存的一部分一样。 程序可用于引用存储器的地址与存储器系统用于识别物理存储站点的地址不同，程序生成的地址自动转换为相应的机器地址。
虚拟存储的大小受到计算机系统的寻址方案的限制，并且辅助存储器的数量不是由主存储位置的实际数量可用。

这是一种使用硬件和软件实现的技术。 它将程序使用的内存地址（称为虚拟地址）映射到计算机内存中的物理地址。

  1. 进程中的所有内存引用都是在运行时动态转换为物理地址的逻辑地址。 这意味着可以将进程交换进出主存储器，使得它在执行过程中的不同时间占用主存储器中的不同位置。
  2. 可以将处理分成多个片段，并且这些片段在执行期间不需要连续地位于主存储器中。 动态运行时地址转换和页面或段表的使用的组合允许这样做。

如果存在这些特性，则在执行期间不必将所有页面或段存在于主存储器中。 这意味着需要时需要将所需页面加载到内存中。 虚拟内存使用Demand Paging或Demand Segmentation实现。

需求寻呼：
按需将页面加载到内存中的过程（每当发生页面错误时）称为请求分页。
该过程包括以下步骤：

  1. 如果CPU尝试引用主存储器中当前不可用的页面，则会生成指示存储器访问故障的中断。
  2. 操作系统将中断的进程置于阻塞状态。 为了继续执行，操作系统必须将所需的页面带入内存。
  3. 操作系统将在逻辑地址空间中搜索所需的页面。
  4. 所需页面将从逻辑地址空间带到物理地址空间。 页面替换算法用于替换物理地址空间中的页面的决策。
  5. 页面表将相应更新。
  6. 信号将被发送到CPU以继续执行程序，它将使进程返回就绪状态。

因此，每当发生页面错误时，操作系统都会遵循这些步骤，并将所需页面带入内存。



优点 ：

  • 主存储器中可以维护更多进程：因为我们只会加载任何特定进程的某些页面，所以还有更多进程的空间。 这导致处理器的更有效利用，因为更可能的是，在任何特定时间，更多的处理中的至少一个将处于就绪状态。
  • 一个过程可能比所有主存储器都要大：编程中最基本的限制之一被取消。 由于需求分页，可以执行大于主存储器的过程。 OS本身根据需要在主存储器中加载进程的页面。
  • 它通过为每个进程使用较少的可用（主）内存来允许更高的多道程序设计级别。

页面错误服务时间：
服务页面错误所需的时间称为页面错误服务时间。 页面错误服务时间包括执行上述所有六个步骤所需的时间。 

 Let Main memory access time is: m
Page fault service time is: s
Page fault rate is : p
Then, Effective memory access time = (p*s) + (1-p)*m

交换：

交换进程意味着从内存中删除所有页面，或者标记它们以便通过正常页面替换过程将其删除。 挂起进程可确保在换出进程时不可运行。 稍后，系统将进程从辅助存储器交换回主存储器。 当进程忙于交换页面时，这种情况称为颠簸。

捶打：

在任何给定时间，任何进程中只有少数页面位于主存储器中，因此可以在存储器中维护更多进程。 此外，节省了时间，因为未使用的页面不会换入和换出内存。 但是，操作系统必须聪明地管理这个方案。 实际上，在稳定状态下，所有主存储器都将被进程页面占用，因此处理器和操作系统可以直接访问尽可能多的进程。 因此，当操作系统带入一页时，它必须抛出另一页。 如果它在使用之前抛出一个页面，那么它几乎必须立即再次获取该页面。 太多这导致了一种叫做Thrashing的情况。 系统花费大部分时间交换页面而不是执行指令。 因此需要一个好的页面替换算法。

在给定的图中，多点编程的初始程度达到某种程度的点（lamda），CPU利用率非常高并且系统资源被100％利用。 但是，如果我们进一步提高多编程的程度，CPU利用率将急剧下降，系统将只在页面替换中花费更多时间，并且完成执行过程所花费的时间将增加。 系统中的这种情况称为颠簸。

颠簸的原因：

  1. 高度多道程序化 ：如果内存中的进程数继续增加，则分配给每个进程的帧数将减少。 因此，每个进程可用的帧数较少。 因此，页面错误将更频繁地发生，并且仅仅交换页面就会浪费更多的CPU时间，并且利用率将继续降低。

    例如：
    让自由帧= 400
    案例1 ：流程数= 100
    然后，每个进程将获得4帧。

    案例2 ：流程数= 400
    每个过程将获得1帧。
    情况2是颠簸的条件，随着处理的数量增加，每个处理的帧减少。 因此，只需交换页面就会消耗CPU时间。

  2. 缺少框架 ：如果一个进程的帧数较少，那么该进程的较少页面将能够驻留在内存中，因此需要更频繁地进行交换。 这可能会导致颠簸。 因此，必须为每个过程分配足够数量的帧以防止颠簸。

颠簸的恢复：

  • 通过指示长期调度程序在阈值之后不将进程置于内存中，不允许系统进入颠簸状态。
  • 如果系统已经在颠簸，那么指示中期计划暂停某些进程，以便我们可以从颠簸中恢复系统。
