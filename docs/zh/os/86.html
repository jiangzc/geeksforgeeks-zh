倒置页表

先决条件 - 分页 ， 页表条目 ， 分段
大多数操作系统为每个进程实现单独的页面表，即对于在多处理/分时操作系统上运行的“n”个进程，存储器中存储有'n'个页表。 有时，当一个进程的大小非常大并且占用虚拟内存时，那么随着进程的大小，它的页面大小也会大大增加。 

 Example: A process of size 2 GB with:
Page size = 512 Bytes
Size of page table entry = 4 Bytes, then
Number of pages in the process = 2 GB / 512 B = 2 22
PageTable Size = 2 22 * 2 2 = 2 24 bytes

通过这个例子，可以得出结论，对于在OS中同时运行的多个进程，相当大一部分内存仅由页表占用。

操作系统还包含多级分页方案 ，这进一步增加了存储页表的空间要求，并且在存储它们时投入了大量内存。 页表占用的内存量可能会成为巨大的开销，并且总是不可接受的，因为主内存始终是稀缺资源。 进行了各种努力以有效地利用存储器并在多编程和有效CPU利用的水平上保持良好的平衡。

倒置页面表 -
另一种方法是使用反向页表结构，该结构由主存储器的每个帧的一个页表条目组成。 因此，倒置页面表中的页表条目数减少到物理内存中的帧数，单个页表用于表示所有进程的分页信息。

通过反向页表，消除了为每个进程存储单个页面表的开销，并且仅需要固定的存储器部分来将所有进程的寻呼信息存储在一起。 该技术被称为反向寻呼，因为关于帧号而不是逻辑页号进行索引。 页表中的每个条目都包含以下字段。



  • 页码 -它指定逻辑地址的页码范围。

  • 进程ID -反向页表包含正在执行的所有进程的地址空间信息。 由于两个不同的进程可以具有相似的虚拟地址集，因此在反向页表中必须存储每个进程的进程ID以唯一地标识它的地址空间。 这是通过使用PId和Page Number的组合来完成的。 因此，此Process Id充当地址空间标识符，并确保特定进程的虚拟页面正确映射到相应的物理帧。

  • 控制位 -这些位用于存储额外的与寻呼相关的信息。 这些包括有效位，脏位，参考位，保护和锁定信息位。

  • 链式指针 -有时可能有两个或多个进程共享主内存的一部分。 在这种情况下，两个或多个逻辑页映射到同一页表项，然后使用链接指针将这些逻辑页的详细信息映射到根页表。

示例 - Inverted Pagetable及其变体在各种系统中实现，如PowerPC，UltraSPARC和IA-64架构。 在RT-PC上实现Mach操作系统也使用这种技术。

的优点和缺点：

  • 减少内存空间 -
    反向页表通常会减少将页表存储到物理内存大小所需的内存量。 最大条目数可以是物理内存中的页面帧数。

  • 查找时间更长 -
    反向页面表按帧编号顺序排序，但内存查找是相对于虚拟地址进行的，因此，查找相应的条目通常需要较长时间，但这些页表通常使用哈希数据结构实现更快抬头。

  • 困难的共享内存实现 -
    由于反转页表为每个帧存储单个条目，因此在页表中实现共享内存变得很困难。 链接技术用于将多个虚拟地址映射到按帧编号指定的条目。
