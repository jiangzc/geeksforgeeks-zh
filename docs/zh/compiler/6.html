编译器设计中的错误处理

错误处理过程的任务是检测每个错误，将其报告给用户，然后制定一些恢复策略并实施它们以处理错误。 在整个过程中，程序的处理时间不应该很慢。 “ 错误”是符号表中的空白条目。

错误的类型或来源 -有两种类型的错误：运行时和编译时错误：

  1. 运行时错误是在程序执行期间发生的错误，通常是由于不利的系统参数或无效的输入数据而发生的。 缺少足够的内存来运行应用程序或内存与另一个程序冲突和逻辑错误就是这种情况的一个例子。 当执行的代码不产生预期结果时发生逻辑错误。 逻辑错误最好通过细致的程序调试来处理。
  2. 在编译程序执行之前，编译时错误会在编译时上升。 阻止程序成功编译的语法错误或缺少文件引用就是一个例子。

编译时错误的分类 -

  1. 词汇 ：这包括标识符，关键字或运算符的拼写错误
  2. 句法 ：缺少分号或不平衡的括号
  3. 语义 ：不兼容的值赋值或运算符与操作数之间的类型不匹配
  4. 逻辑 ：代码无法访问，无限循环。

查找错误或报告错误 - Viable-prefix是解析器的属性，允许及早检测语法错误。

  • 目标：尽快检测错误，而无需进一步消耗不必要的输入
  • 方法：一旦输入的前缀与中的任何字符串的前缀不匹配，就会检测到错误
    语言。
  • 示例： for（ ; ），这将报告错误，因为在大括号内有两个分号。

错误恢复 -
编译器的基本要求是简单地停止和发出消息，并停止编译。 下面是一些常见的恢复方法。

  1. 紧急模式恢复：这是最简单的错误恢复方式，并且它还可以防止解析器在恢复错误时开发无限循环。 解析器一次丢弃一个输入符号，直到找到一组指定的（如结束，分号）同步令牌（通常是语句或表达式终止符）。 当同一语句中存在多个错误很少时，这就足够了。 示例：考虑错误的表达式 - （1 + + 2）+ 3.恐慌模式恢复：向前跳到下一个整数然后继续。 Bison：使用特殊终端错误来描述要跳过的输入量。 
     E->int|E+E|(E)|error int|(error)
  2. 相位级恢复：对输入执行本地校正以修复错误。 但是这种策略很难纠错。
  3. 错误产生：编译器设计者可能知道代码中可能出现的一些常见错误。 也可以使用增强语法作为在遇到这些错误时生成错误结构的产品。 示例：写入5x而不是5 * x
  4. 全局更正：其目的是在将不正确的输入字符串转换为有效字符串时尽可能少地进行更改。 实施该策略的成本很高。
