流水线处理器中的依赖关系

流水线处理器中可能存在三种类型的依赖关系。 这些是 ：
1）结构依赖性
2）控制依赖性
3）数据依赖性

这些依赖关系可能会在管道中引入停顿。

失速：失速是管道中没有新输入的循环。

结构依赖

由于管道中的资源冲突，产生了这种依赖性。 资源冲突是指多个指令尝试在同一周期中访问同一资源的情况。 资源可以是寄存器，内存或ALU。



例：

指令/周期 1 2 3 4 五
我1 IF（MEM） ID EX 纪念品
我2 IF（MEM） ID EX
我3 IF（MEM） ID EX
我4 IF（MEM） ID

在上面的场景中，在周期4中，指令I 1和I 4试图访问引入资源冲突的相同资源（存储器）。
为了避免这个问题，我们必须让指令保持等待，直到所需的资源（在我们的例子中为内存）可用。 此等待将在管道中引入停顿，如下所示：

周期 1 2 3 4 五 6 7 8
我1 IF（MEM） ID EX 纪念品 WB
我2 IF（MEM） ID EX 纪念品 WB
我3 IF（MEM） ID EX 纪念品 WB
我4 - - - IF（MEM）


结构依赖的解决方案
为了最小化管道中的结构依赖性停滞，我们使用称为重命名的硬件机制。
重命名：根据重命名，我们将存储器分成两个独立的模块，分别用于存储指令和数据，分别称为代码存储器（CM）和数据存储器（DM）。 CM将包含所有指令，DM将包含指令所需的所有操作数。

指令/周期 1 2 3 4 五 6 7
我1 IF（CM） ID EX DM WB
我2 IF（CM） ID EX DM WB
我3 IF（CM） ID EX DM WB
我4 IF（CM） ID EX DM
我5 IF（CM） ID EX
我6 IF（CM） ID
我7 IF（CM）



控制依赖（分支危险）
这种类型的依赖性发生在诸如BRANCH，CALL，JMP等控制指令的传送期间。在许多指令体系结构中，当处理器需要将新指令插入流水线时，处理器将不知道这些指令的目标地址。 因此，不需要的指令被馈送到管道。

考虑程序中的以下指令序列：
100：我1
101：I 2 （JMP 250）
102：我3
。
。
250：BI 1

预期输出：I 1 - > I 2 - > BI 1

注意：通常，JMP指令的目标地址仅在ID阶段后才知道。

指令/周期 1 2 3 4 五 6
我1 如果 ID EX MEM WB
我2 如果 ID（PC：250） EX 纪念品 WB
我3 如果 ID EX 纪念品
BI 1 如果 ID EX

输出序列：I 1 - > I 2 - > I 3 - > BI 1

因此，输出序列不等于预期输出，这意味着管道未正确实现。

为了纠正上述问题，我们需要停止指令获取，直到得到分支指令的目标地址。 这可以通过引入延迟槽来实现，直到我们得到目标地址。

指令/周期 1 2 3 4 五 6
我1 如果 ID EX MEM WB
我2 如果 ID（PC：250） EX 纪念品 WB
延迟 - - - - - -
BI 1 如果 ID EX

输出序列：I 1 - > I 2 - >延迟（失速） - > BI 1

由于延迟槽不执行任何操作，因此该输出序列等于预期的输出序列。 但是这个插槽引入了管道中的失速。

分支惩罚：在流水线处理器中的分支操作期间引入的停顿数称为分支惩罚。

注意：正如我们在ID阶段之后看到目标地址可用，因此管道中引入的停顿数是1.假设，在ALU阶段之后，分支目标地址将存在，那将有2个停顿。 通常，如果在第k 个阶段之后存在目标地址，则管道中将存在（k-1）个停顿。

由于分支指令导致管道中引入的停顿总数= 分支频率*分支惩罚


数据依赖（数据危害）
让我们考虑一个ADD指令S.
S：添加R1，R2，R3
读取的地址S = I（S）= {R2，R3}
由S = O（S）= {R1}写的地址

现在，我们说指令S2取决于指令S1，何时


这种情况称为伯恩斯坦条件。

存在三种情况：

  • 流（数据）依赖性：O（S1）∩I（S2），S1→S2和S1在S2读取之后写入
  • 反依赖性：I（S1）∩O（S2），S1→S2和S1在S2覆盖之前读取内容
  • 输出依赖性：O（S1）∩O（S2），S1→S2并且都写入相同的存储位置。

示例：让I1和I2有两条指令：
I1：添加R1，R2，R3
I2：SUB R4，R1，R2

当在流水线处理器中执行上述指令时，将发生数据依赖性条件，这意味着I 2在I 1写入之前尝试读取数据，因此，I 2错误地从I 1获取旧值。

指令/周期 1 2 3 4
我1 如果 ID EX DM
我2 如果 ID（旧值） EX

为了最小化管道中的数据依赖性停顿，使用操作数转发 。

操作数转发：在操作数转发中，我们使用各级之间的接口寄存器来保存中间输出，以便相关指令可以直接从接口寄存器访问新值。

考虑相同的例子：
I1：添加R1，R2，R3
I2：SUB R4，R1，R2

指令/周期 1 2 3 4
我1 如果 ID EX DM
我2 如果 ID EX



数据危害

当表现出数据依赖性的指令在管道的不同阶段修改数据时，就会发生数据危险。 危害导致管道延误。 主要有三种类型的数据危害：

1）RAW（写入后读取）[流量依赖性]
2）WAR（读后写）[反数据依赖]
3）WAW（写入后写入）[输出依赖性]

让I和J有两个指令，这样J跟随我。然后，

  • 当指令J在写入指令之前尝试读取数据时，会发生RAW危险。
    例如：
    I：R2 < - R1 + R3
    J：R4 < - R2 + R3
  • 当指令J在指令I读取数据之前尝试写入数据时，会发生WAR危险。
    例如：
    I：R2 < - R1 + R3
    J：R3 < - R4 + R5
  • 当指令J在写入指令之前尝试写入输出时，会发生WAW危险。
    例如：
    I：R2 < - R1 + R3
    J：R2 < - R4 + R5

在无序执行指令期间发生WAR和WAW危险。
