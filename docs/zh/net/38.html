滑动窗口协议（发送方）| 设置1

滑动窗口协议实际上是一个理论概念，我们只讨论了发送方窗口大小（1 + 2a）应该是什么，以便提高停止和等待arq的效率。 现在我们将讨论实际的实现，我们将在其中处理接收器窗口的大小。 实际上它是用两种协议实现的，即：

1.返回N（GBN）
2.选择性重复（SR）

在本文中，我们将根据三个主要特征特征向您解释第一个GBN协议，最后一部分我们将讨论SR以及这两个协议的比较

发件人窗口大小（WS）
它本身就是N. 如果我们说协议是GB10，那么Ws = 10.N应该总是大于1才能实现流水线操作。 对于N = 1，它将减少为停止和等待协议。



GBN的效率= N /（1 + 2a）其中a = Tp / Tt

如果B是信道的带宽，则有效带宽或吞吐量=效率*带宽=（N /（1 + 2a））* B.

接收器窗口大小（WR）
WR总是1 GBN。

现在在GBN中究竟发生了什么，我们将在示例的帮助下进行解释。 考虑下面给出的图表。 我们的发送方窗口大小为4.假设我们有很多序列号只是为了解释。 现在发送方已经发送了分组0,1,2和3.在确认分组0和1之后，接收机现在正在期望分组2并且发送方窗口也滑动以进一步发送分组4和5.现在假设分组2是丢失在网络中，Receiver将丢弃发送方在数据包2之后发送的所有数据包，因为它预期序列号为2.在发送方，每个数据包发送都有一个超时定时器，它将在2号数据包到期。从最后一个发送的数据包5发送器将返回到当前窗口中的数据包编号2并将所有数据包发送到数据包编号5.这就是为什么它被称为返回N.返回意味着发送者必须返回N个地方最后发送的数据包在未确认的窗口中，而不是从数据包丢失的点开始。

致谢
有两种确认即：

  • 累积确认 - 一个确认用于许多数据包。 主要优点是交通量较少。 缺点是可靠性较低，好像一个ack是丢失，这意味着所有发送的数据包都丢失了。
  • 独立确认 - 如果每个数据包都将独立获得确认。 这里的可靠性很高，但缺点是流量也很高，因为对于我们接收的每个数据包都是独立的ack。

GBN使用累积确认。 在接收器端，每当接收器接收到任何固定的数据包时，它就启动一个确认定时器，当它到期时，它将发送一个累积Ack，用于在该定时器间隔内接收的数据包数。 如果接收方已收到N个数据包，则确认号码将为N + 1。 重点是确认定时器在第一个定时器到期后但在接收器收到数据包后才会启动。
发送方的超时定时器应大于确认定时器。

窗口大小与序列号之间的关系
我们已经知道所需的序列号应始终等于任何滑动窗口协议中的窗口大小。

考虑一下GB4的一个例子。 发件人窗口大小为4因此我们需要至少4个序列号来标记窗口中的每个数据包。 现在假设接收方已收到所有数据包（发送方发送的0,1,2和3），因此现在再次等待数据包0（我们不能在这里使用4，因为我们只有4个序列号，因为N = 4） 。 现在假设上述4个数据包的累积ack在网络中丢失。 在发送方，数据包0将超时，因此所有4个数据包将再次传输。 问题现在是接收器正在等待应该从0开始的新数据包集，但现在它将接收先前接受的数据包的重复副本。 为了避免这种情况，我们需要一个额外的序列号。 现在接收可以很容易地拒绝从0开始的所有重复数据包，因为现在它将等待4号数据包（我们现在已经添加了一个额外的序列号）。
