TCP定时器

TCP使用多个定时器来确保在通信期间不会遇到过多的延迟。 其中一些计时器很优雅，处理的问题在初次分析时并不是很明显。 TCP使用的每个定时器将在以下部分中进行检查，这些部分揭示了它在确保数据从一个连接正确发送到另一个连接方面的作用。

TCP实施使用四个定时器 -

  1. 重传定时器 -为了重传丢失的分段，TCP使用重传超时（RTO）。 当TCP发送一个段时，定时器启动并在收到确认时停止。 如果计时器到期，则发生超时并重新传输该段。 RTO（重传超时为1 RTT）计算重传超时，我们首先需要计算RTT（往返时间）。
    RTT三种类型 -

    • 测量的RTT（RTTm） -测量的段的往返时间是段到达目的地并被确认所需的时间，尽管确认可能包括其他段。
    • 平滑RTT（RTT） -它是RTTm的加权平均值。 RTTm可能会发生变化，其波动很大，单个测量不能用于计算RTO。 
       Initially -> No value
After the first measurement -> RTTs=RTTm
After each measurement -> RTTs= (1-t)*RTTs + t*RTTm
Note: t=1/8 (default if not given)
    • 偏差RTT（RTTd） -大多数实现不单独使用RTT，因此也计算RTT偏差以找出RTO。 
       Initially -> No value
After the first measurement -> RTTd=RTTm/2
After each measurement -> RTTd= (1-k)*RTTd + k*(RTTm-RTTs)
Note: k=1/4 (default if not given)

    重传超时：RTO计算 - RTO的值基于平滑的往返时间及其偏差。 大多数实现使用以下公式来计算RTO： 

     Initial value -> Original (given in question)
After any measurement -> RTO=RTTs + 4*RTTd

    #NOTE：每次重传时，RTO的值都会翻倍。 （RTO（新）= RTO（重传之前）* 2）这在Karn算法中有解释

  2. 持久定时器 -为了处理零窗口大小的死锁情况，TCP使用持久性定时器。当发送TCP收到窗口大小为零的确认时，它启动持久性定时器。 当持久性计时器关闭时，发送TCP发送一个称为探测的特殊段。 该段仅包含1个字节的新数据。 它有一个序列号，但它的序列号永远不会被确认; 在计算其余数据的序列号时甚至会忽略它。 探测器使接收TCP重新发送丢失的确认。
  3. 保持活动计时器 - 保持活动计时器用于防止两个TCP之间的长时间空闲连接。 如果客户端打开到服务器的TCP连接，则传输一些数据并变为静默，客户端将崩溃。 在这种情况下，连接将永远保持打开状态。 因此使用了keepalive计时器。 每次服务器从客户端听到时，它都会重置此计时器。 超时通常为2小时。 如果服务器在2小时后没有收到来自客户端的消息，则会发送探测段。如果10个探测器之后没有响应，每个探测器相隔75秒，则假定客户端已关闭并终止连接。
  4. 时间等待计时器 -在tcp连接终止期间使用此计时器。 在发送第二个FIN的最后一个Ack并关闭连接后，计时器启动。
