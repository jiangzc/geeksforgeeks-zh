滑动窗口协议| 设置1（发件人侧）

先决条件： 停止并等待ARQ

停止和等待ARQ提供错误和流量控制，但可能会导致严重的性能问题，因为发送方始终等待确认，即使它已准备好发送下一个数据包。 考虑一种情况，即你有一个高带宽连接，传播延迟也很高（你通过高速连接连接到某个其他国家的某个服务器），由于停止和等待的限制，你不能使用这个全速。

Sliding Window协议通过一次发送多个包含较大序列号的数据包来处理此效率问题。 这个想法与体系结构中的流水线相同。

几个术语：

传输延迟（Tt） - 从主机向传出链路传输数据包的时间。 如果B是链路的带宽，D是要传输的数据大小 

 Tt = D/B

传播延迟（Tp） - 主机传输到传出链路的第一个比特到达目的地所用的时间。 它取决于距离d和波传播速度s（取决于介质的特性）。 



 Tp = d/s

效率 - 定义为总有用时间与数据包总周期时间的比率。 对于停止和等待协议， 

 Total cycle time = Tt(data) + Tp(data) + 
                    Tt(acknowledgement) + Tp(acknowledgement)
 	         = Tt(data) + Tp(data) + Tp(acknowledgement)
		 = Tt + 2*Tp

由于确认的大小非常小，因此可以忽略它们的传输延迟。 

 Efficiency = Useful Time / Total Cycle Time 
           = Tt/(Tt + 2*Tp) (For Stop and Wait)
           = 1/(1+2a) [ Using a = Tp/Tt ]

有效带宽（EB）或吞吐量 - 每秒发送的位数。 

 EB = Data Size(L) / Total Cycle time(Tt + 2*Tp)
Multiplying and dividing by Bandwidth (B),
       = (1/(1+2a)) * B [ Using a = Tp/Tt ]
       = Efficiency * Bandwidth

链路容量 - 如果信道是全双工，则可以在两个方向上传输比特，而不会发生任何冲突。 通道/链路最大可容纳的位数是其容量。 

 Capacity = Bandwidth(B) * Propagation(Tp)
	    
 For Full Duplex channels, 
 Capacity = 2*Bandwidth(B) * Propagation(Tp)

流水线的概念

在停止和等待协议中，只有1个数据包被传输到链路上，然后发送器等待来自接收器的确认。 这种设置中的问题是效率非常低，因为我们没有在第一个数据包被放到链路上之后用更多的数据包填充信道。 在Tt + 2 * Tp单位的总周期时间内，我们现在将计算发送者在获得确认之前可以在链路上传输的最大数据包数。 

 In Tt units ----> 1 packet is Transmitted.
 In 1 units ----> 1/Tt packet can be Transmitted.
 In Tt + 2*Tp units -----> (Tt + 2*Tp)/Tt 
                             packets can be Transmitted
	            ------> 1 + 2a [Using a = Tp/Tt]

可以在总周期时间内发送的最大数据包数= 1 + 2 * a

发件人窗口的最小位数（非常重要的GATE）

如上所述， 

 Maximum window size = 1 + 2*a where a = Tp/Tt

 Minimum sequence numbers required = 1 + 2*a.

当前窗口中的所有数据包都将被赋予序列号。 表示发送方窗口所需的位数= ceil（log2（1 + 2 * a））。

但有时协议头中的位数是预先定义的。 标头中序列号字段的大小也将决定我们可以在总循环时间内发送的最大数据包数。 如果N是标题中的序列号字段的大小，则可以有2个N序列号。

窗口尺寸ws = min（1 + 2 * a，2 N ）

如果要计算表示序列号/发送方窗口所需的最小位数， 则为ceil（log2（ws）） 。
