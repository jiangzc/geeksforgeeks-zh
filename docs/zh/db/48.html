DBMS中的ACID属性

事务是单个逻辑工作单元，它访问并可能修改数据库的内容。 事务使用读写操作访问数据。
为了在事务之前和之后保持数据库的一致性，遵循某些属性。 这些被称为ACID属性。

原子性
通过这种方式，我们的意思是整个交易要么立刻发生，要么根本不发生。 没有中途，即交易不会部分发生。 每个事务都被视为一个单元，并且运行完成或根本不执行。 它涉及以下两个操作。
- 中止 ：如果事务中止，则对数据库所做的更改不可见。
- 提交 ：如果事务提交，则所做的更改是可见的。
原子性也被称为“全有或全无规则”。
考虑以下由T1和T2组成的交易T ：从账户X到账户Y的100转账。

如果交易在T1完成后但在T2完成之前失败（例如，在写入（X）之后但在写入（Y）之前），则从X中扣除金额但不将其添加到Y. 这会导致数据库状态不一致。 因此，必须完整地执行事务以确保数据库状态的正确性。


一致性
这意味着必须保持完整性约束，以便数据库在事务之前和之后保持一致。 它指的是数据库的正确性。 参考上面的例子，
必须保持交易前后的总金额。
T发生前的总数= 500 + 200 = 700 。
T发生后的总数= 400 + 300 = 700 。
因此，数据库是一致的 。 在T1完成但T2失败的情况下发生不一致。 结果T不完整。




隔离
此属性确保可以同时发生多个事务，而不会导致数据库状态不一致。 交易独立发生，不受干扰。 在将该事务中的特定更改写入内存或已提交之前，任何其他事务都不会看到特定事务​​中发生的更改。 此属性确保并发执行事务将导致状态等效于以某种顺序串行执行的状态。
设X = 500， Y = 500。
考虑两个事务T和T“。

假设T已执行直到读（Y）然后T''开始。 结果，由于T''读取X的正确值但是Y的值不正确并且由
T'':( X + Y = 50,000 + 500 = 50,500）
因此与交易结束时的总和不一致：
T：（X + Y = 50,000，450 = 50,450） 。
由于损失了50个单位，这导致数据库不一致。 因此，事务必须隔离进行，并且只有在对主存储器进行更改后才能看到更改。


耐用性：
此属性确保一旦事务完成执行，对数据库的更新和修改将存储在磁盘中并写入磁盘，即使系统发生故障也会持续存在。 这些更新现在变为永久更新并存储在非易失性存储器中。 因此，交易的影响永远不会丢失。

总体而言， ACID属性提供了一种机制，以确保数据库的正确性和一致性，使得每个事务都是一组操作，它们作用于一个单元，产生一致的结果，与其他操作隔离，并更新它品牌经久耐用。
