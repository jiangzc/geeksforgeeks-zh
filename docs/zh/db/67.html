DBMS中的饥饿

当事务必须等待一段不确定的时间来获取锁时，就会出现Starvation或Livelock。

饥饿的原因 -

  • 如果等待锁定物品的计划是不公平的。 （优先队列）
  • 受害者选择。 （重复选择同一交易作为受害者）
  • 资源泄漏。
  • 通过拒绝服务攻击。

在示例的帮助下可以最好地解释饥饿 - 假设数据库中有3个事务，即T1，T2和T3，它们试图获取对数据项“I”的锁定。 现在，假设调度程序将锁定授予T1（可能是由于某些优先级），而其他两个事务正在等待锁定。 一旦T1的执行结束，另一个事务T4也会过来并在数据项I上请求解锁。现在，这次调度器授予对T4的锁定，并且T2，T3必须再次等待。 这样，如果新的交易继续请求锁定，则T2和T3可能必须等待不定的时间段，这导致饥饿 。

饥饿的解决方案是什么 -

  1. 提高优先级 -
    当事务必须等待不确定的时间时发生饥饿。在这种情况下，我们可以增加该特定事务的优先级。 但是这种解决方案的缺点是可能发生另一个事务可能需要等待更长时间，直到最高优先级事务到来并继续。
  2. 受害者选择算法中的修改 -
    如果事务是重复选择的受害者，则可以通过降低其优先于其他事务的优先级来修改算法。
  3. 先到先得的服务方法 -
    可以采用公平的调度方法，即FCFS，其中事务可以获取对订单中的Item的锁定，其中请求锁定。
  4. 等待死亡和伤口等待计划 -
    这些是使用事务的时间戳排序机制的方案。
