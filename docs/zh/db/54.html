并发控制协议| 两相锁定（2-PL）-III

先决条件 - 两相锁定协议（2-PL）的基础知识，2-PL的 类型 。

现在，我们知道Strict 2-PL和Rigorous 2-PL都避免了级联回滚，并确保严格的时间表，但仍然无法保证我们的时间表无死锁。 我们已经看到严格和严格的2-PL在应用上是相似的，并且一般的误解是常见的，保守2-PL也遵循与上述两个相同的协议集。 为清楚起见，我们详细介绍一下Conservative 2-PL。

保守的2-PL -

AKA Static 2-PL ，此协议要求事务在事先开始执行之前锁定它访问的所有项目，方法是预先设置其读取集和写入集。 如果无法锁定任何所需的预先声明的项目，则事务不会锁定任何项目，而是等待所有项目都可用于锁定。 因此，在我们锁定所需的所有项目之前，对数据的操作无法启动。

现在让我们看一个关于保守党2-PL的有趣例子。 告诉我以下时间表是否遵循保守2-PL？ 

 Schedule: Lock-S(A) Lock-X(B) Read(A) Read(B) Write(A) Unlock(A) Commit Unlock(B)

你认为上述附表不遵循保守派2-PL吗？ 不要将协议混淆为Rigorous 2-PL的修改版本，我们可以随时释放锁，但我们需要在执行任何操作之前锁定所有数据项。 这就是它无死锁的原因。 上述时间表遵循保守党2-PL。
保守派2-PL的一些有趣特征：



  • 我们在Basic，Strict和Rigorous 2-PL中看到的后续时间表将没有成长阶段。 由于在使用数据之前锁定数据是强制性的，因此该协议没有生长阶段 。 此外，此规则使其无死锁，就好像一个项目不可用于锁定事务释放所有锁定并稍后再次尝试，即没有保持和等待 。 这使得死锁的四个必要条件之一无效。
  • 我们只需事先锁定所有项目，因此释放或解锁它们没有像我们在Strict或Rigorous 2-PL中那样的限制。
  • 由于在获取所有锁之前没有进行任何操作，因此与Basic，Strict，Rigorous 2-PL不同，我们在此协议中没有生长阶段。
  • 虽然我们得到了一个无死锁的时间表，但在这个协议中我们仍然可能面临像Cascading Rollbacks这样的缺点。 所以这个协议不能确保严格的时间表 。 与Strict和Rigorous 2-PL相比，这是一个缺点。
