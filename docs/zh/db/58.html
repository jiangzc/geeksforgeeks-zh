并发控制协议| 时间戳订购协议

并发控制可以以不同的方式实现 。 实现它的一种方法是使用Locks 。 现在，让我们讨论时间戳订购协议。

如前所述， Timestamp是DBMS创建的唯一标识符，用于标识事务。 它们通常按照提交给系统的顺序分配。 将事务T的时间戳称为TS（T） 。 有关时间戳的基础知识，请参阅此处 。

时间戳订购协议 -

该协议的主要思想是根据其时间戳对事务进行排序。 然后，事务参与的计划可序列化，并且允许的唯一等效序列计划使事务按其时间戳值的顺序排列。 简单地说，时间表等同于与事务时间戳 顺序相对应的特定序列顺序 。 算法必须确保对于计划中冲突操作访问的每个项目，访问项目的顺序不违反排序。 要确保这一点，请使用与每个数据库项X相关的两个时间戳值。

  • W_TS（X）是成功执行write（X）的任何事务的最大时间戳。



  • R_TS（X）是成功执行read（X）的任何事务的最大时间戳。

基本时间戳订购 -

每个事务都根据它进入系统的时间发出时间戳。 假设，如果旧事务T i具有时间戳TS（T i ），则为新事务T j分配时间戳TS（T j ），使得TS（T i ）<TS（T j ） 。该协议管理并发执行，使得时间戳确定可序列化顺序。 时间戳排序协议确保以时间戳顺序执行任何冲突的读写操作。 每当某个事务T尝试发出R_item（X）或W_item（X）时，Basic TO算法会将T的时间戳与R_TS（X）和W_TS（X）进行比较，以确保不违反时间戳顺序。 这描述了以下两种情况下的Basic TO协议。

  1. 每当Transaction T发出W_item（X）操作时，请检查以下条件：

    • 如果R_TS（X）> TS（T）或W_TS（X）> TS（T） ，则中止并回滚T并拒绝该操作。 其他，

    • 执行T的W_item（X）操作并将W_TS（X）设置为TS（T）。
  2. 每当Transaction T发出R_item（X）操作时，请检查以下条件：

    • 如果W_TS（X）> TS（T） ，则中止并拒绝T并拒绝操作，否则

    • 如果W_TS（X）<= TS（T），则执行T的R_item（X）操作，并将R_TS（X）设置为TS（T）和当前R_TS（X）中的较大者。

每当Basic TO算法检测到以错误顺序发生的twp冲突操作时，它会通过中止发出它的事务来拒绝两个操作中的后一个操作。 Basic TO生成的计划保证是冲突可序列化的 。 已经讨论过使用Timestamp，可以确保我们的日程安排无死锁 。

Basic TO协议的一个缺点是它仍然可以进行级联回滚 。 假设我们有一个事务T 1，而T 2使用了由T 1写的值。 如果T 1被中止并重新提交给系统，那么T也必须被中止并回滚。 因此，级联中止的问题仍然存在。

严格的时间戳订购 -

Basic TO的变体称为Strict TO，可确保计划既可以是严格的，也可以是冲突可序列化的。 在此变体中，事务T发出R_item（X）或W_item（X），使得TS（T）> W_TS（X）的读取或写入操作延迟，直到写入X的值的事务T '已提交或中止。
