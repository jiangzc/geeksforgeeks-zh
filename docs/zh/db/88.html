1.鉴于这种关系

员工（姓名，工资，deptno）和
部门（deptno，deptname，address）

使用基本关系代数无法表达以下哪些查询
操作（U， - ，x，π，σ，p）？ （GATE CS 2000）
（a）每名雇员的部门地址
（b）名称与其部门名称相同的雇员
（c）所有雇员工资的总和
（d）特定部门的所有雇员

答案： （c）



说明：
关系代数的六个基本算子是选择（σ），投影（π），笛卡尔乘积（x）（也称为叉积或交叉连接），集合联合（U），集合差异（ - ） ，以及重命名（p）。 这六个运算符是基本的，因为在不失去表达能力的情况下，它们都不能被省略。 许多其他运营商已根据这六个进行了定义。 其中最重要的是设置交集，除法和自然连接，但这些基本关系代数运算无法进行聚合。 因此，我们不能用六项业务来计算所有员工的工资总和。

2.给出以下关系实例。 

 xyz 
 1 4 2 
 1 5 3 
 1 6 3 
 3 2 2

实例满足以下哪些功能依赖性？ （GATE CS 2000）
（a）XY - > Z和Z - > Y.
（b）YZ - > X和Y - > Z.
（c）YZ - > X和X - > Z.
（d）XZ - > Y和Y - > X.

答案： （b）

说明：
功能依赖（FD）是来自数据库的关系中的两组属性之间的约束。 FD X-> Y要求X的值唯一地确定Y的值，其中X和Y是属性集。 FD是关键概念的概括。

假设X，Y和Z是关系R中的属性集，则可以导出函数依赖性的若干属性。 其中最重要的是阿姆斯特朗的公理，它们用于数据库规范化： 

    
* Subset Property (Axiom of Reflexivity): If Y is a subset of X, then X ? Y
* Augmentation (Axiom of Augmentation): If X -> Y, then XZ -> YZ
* Transitivity (Axiom of Transitivity): If X -> Y and Y -> Z, then X -> Z

根据这些规则，我们可以推导出这些次要规则： 

    
* Union: If X -> Y and X -> Z, then X -> YZ
* Decomposition: If X -> YZ, then X -> Y and X -> Z
* Pseudotransitivity: If X -> Y and YZ -> W, then XZ -> W

在上面的问题中，Y唯一地确定X和Z，对于给定的Y值，您可以很容易地找到X和Z的值。
因此，Y - > X和Y - > Z适用于上述模式。
从增强规则我们可以说YZ-> X. 如果我们理解FD的概念，我们不需要应用公理来找出哪个选项是正确的，只需查看我们可以说的模式和选项（b）是真的。

给定关系r（w，x）和s（y，z），得到的结果
选择不同的w，x
来自r，s
保证与r相同，提供（GATE CS 2000）
（a）r没有重复，s是非空的
（b）r和s没有重复
（c）s没有重复，r是非空的
（d）r和s具有相同数量的元组

答案： （a）

说明：
该查询选择r的所有属性。 由于我们在查询中有不同的结果，只有当r没有重复时，结果才能等于r。

如果我们没有给出我们想要连接两个表的任何属性，那么上面的查询就等同于笛卡尔积 。 如果两组中的任何一组为空，则两组的Cartisian产品将为空。 因此，s应至少有一条记录来获取r的所有行。

5.考虑模式R（A，B，C，D）和函数依赖性A - > B和C - > D.然后将R分解为R1（A，B）和R2（C，D）为（GATE） CS 2001）
a）依赖保留和损失减少加入
b）损失少连接但不依赖保留
c）依赖保留但不减少损失加入
d）不依赖保留而不是损失减少加入

答案： （c）

说明：
依赖性保留分解：
如果在分解之后功能依赖性的闭合与在分解之前的FD的闭合相同，则将R分解为R1和R2是依赖性保持分解。
一种简单的方法是检查我们是否可以从分解后存在的FD中导出所有原始FD。

在上面的问题中，R（A，B，C，D）被分解为R1（A，B）和R2（C，D），并且只有两个FD A - > B和C - > D.所以，分解是依赖保留

无损加入分解：
如果以下函数依赖关系中的至少一个在F +中，则将R分解为R1和R2是无损连接分解（功能依赖关闭） 

 R1 ∩ R2 → R1
   OR
    R1 ∩ R2 → R2

在上述问题中，R（A，B，C，D）被分解为R1（A，B）和R2（C，D），并且R1∩R2为空。 因此，分解不是无损的。
