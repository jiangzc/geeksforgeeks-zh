使用功能依赖关系查找属性关闭和候选键

什么是功能依赖？

如果具有相同X值的两个元组也具有相同的Y值，即X唯一地确定Y，则关系中的函数依赖性X-> Y成立。

在表1中给出的EMPLOYEE关系中，

  • FD E-ID-> E-NAME成立，因为对于每个E-ID，存在E-NAME的唯一值。
  • FD E-ID-> E-CITY和E-CITY-> E-STATE也有。
  • FD E-NAME-> E-ID 不成立，因为E-NAME'John'不是唯一确定E-ID的。 有2个E-ID对应John（E001和E003）。

平凡与非平凡的功能依赖：一个微不足道的功能依赖是一个永远存在于关系中的依赖。 

 X->Y will always hold if X ⊇ Y

在上面给出的示例中， E-ID，E-NAME-> E-ID是一个微不足道的功能依赖，并且将始终保持，因为{E-ID，E-NAME}⊃{E-ID}。 您还可以从表中看到，对于{E-ID，E-NAME}的每个值，E-ID的值是唯一的，因此{E-ID，E-NAME}在功能上确定E-ID。

如果函数依赖不是微不足道的，那么它被称为非平凡功能依赖 。 非平凡的功能依赖可能存在也可能不存在于关系中。 例如; E-ID-> E-NAME是一种非平凡的功能依赖，它具有上述关系。

功能依赖性的属性

设X ， Y和Z是关系R中的属性集。 功能依赖的几个属性总是存在于R中，也称为Armstrong Axioms。

  1. 反身性 ：如果Y是X的子集，那么X → Y.例如; 设X表示{E-ID，E-NAME}，Y表示{E-ID}。 {E-ID，E-NAME} - >关系的E-ID为真。
  2. 增强 ：如果X → Y ，则XZ → YZ。 例如; 设X表示{E-ID}，Y表示{E-NAME}，Z表示{E-CITY}。 由于{E-ID} - > E-NAME对于关系是真的，因此{E-ID，E-CITY} - > {E-NAME，E-CITY}也将成立。
  3. 传递性 ：如果X → Y和Y → Z ，则X → Z 。 设X表示{E-ID}，Y表示{E-CITY}，Z表示{E-STATE}。 由于{E-ID} - > {E-CITY}和{E-CITY} - > {E-STATE}对于关系为真，因此{E-ID} - > {E-STATE}也将为真。
  4. 属性闭包：功能上依赖于属性A的属性集称为A的属性闭包，它可以表示为A + 。

找到A的属性闭包的步骤

Q.给定Relation R的FD集，属性闭包集S是A的集合

  1. 添加A到S.
  2. 递归地添加可以从集合S的属性在功能上确定的属性，直到完成为止。

候选钥匙

候选键是关系的最小属性集，可用于唯一地标识元组。 例如，表1中给出的每个EMPLOYEE关系元组都可以通过E-ID唯一标识，并且它也是最小的。 所以它将成为关系的候选关键。

候选键可以是也可以不是主键。

超级钥匙

超级密钥是一组关系的属性 ，可用于唯一地标识元组。例如，表1中给出的每个EMPLOYEE关系元组可以通过E-ID或（E-ID，E-NAME）唯一标识或（E-ID，E-CITY）或（E-ID，E-STATE）或（E_ID，E-NAME，E-STATE）等。因此所有这些都是EMPLOYEE关系的超级密钥。 

 Note: A candidate key is always a super key but vice versa is not true. 
  Q.使用FD集查找关系的候选键和超级键关系的所有属性的属性闭包的属性集称为关系的超级键。 例如，表1中显示的EMPLOYEE关系具有以下FD集。  {E-ID-> E-NAME，E-ID-> E-CITY，E-ID-> E-STATE，E-CITY-> E-STATE}让我们计算不同属性集的属性闭包： 
 (E-ID) + = {E-ID, E-NAME,E-CITY,E-STATE} 
 (E-ID,E-NAME) + = {E-ID, E-NAME,E-CITY,E-STATE} 
 (E-ID,E-CITY) + = {E-ID, E-NAME,E-CITY,E-STATE} 
 (E-ID,E-STATE) + = {E-ID, E-NAME,E-CITY,E-STATE} 
 (E-ID,E-CITY,E-STATE) + = {E-ID, E-NAME,E-CITY,E-STATE} 
 (E-NAME) + = {E-NAME} 
 (E-CITY) + = {E-CITY,E-STATE} 
 作为（E-ID） + ，（E-ID，E-NAME） + ，（E-ID，E-CITY） + ，（E-ID，E-STATE） + ，（E-ID，E-CITY， E-STATE） +给出关系EMPLOYEE的所有属性的集合。 所以这些都是关系的超级钥匙。 
 属性闭包设置为关系的所有属性的最小属性集称为关系的候选关键字。 如上所示，（E-ID） +是关系的所有属性的集合，并且它是最小的。 因此，E-ID将成为候选密钥。 另一方面（E-ID，E-NAME） +也是所有属性的集合，但它不是最小的，因为它的子集（E-ID） +等于所有属性的集合。 因此（E-ID，E-NAME）不是候选密钥。
