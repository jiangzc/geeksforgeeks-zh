并发控制协议| 多粒度锁定

先决条件 - 时间戳订购协议
在各种并发控制方案中，使用了不同的方法和每个单独的数据项作为执行同步的单元。 该技术的某个缺点是如果事务T i需要访问整个数据库，并且使用锁定协议，则T i必须锁定数据库中的每个项目。 效率较低，如果T i可以使用单个锁来锁定整个数据库，则会更简单。 但是，如果考虑第二个提案，实际上这不应该忽略提出的方法中的某些缺陷。 假设另一个事务只需要从数据库访问一些数据项，因此锁定整个数据库似乎是不必要的，而且可能会导致我们失去并发性，这首先是我们的主要目标。 在效率和并发之间讨价还价。 使用粒度。

让我们从了解Granularity的含义开始。

粒度 -允许锁定的数据项的大小。 现在， Multiple Granularity意味着将数据库分层分解为可以锁定的块，并且可以跟踪需要锁定的内容以及以何种方式锁定。 这种层次结构可以图形方式表示为树。

例如，考虑树，它由四个级别的节点组成。 最高级别代表整个数据库。 下面是类型区域的节点; 数据库正好包含这些区域。 区域有子节点，称为文件。 每个区域都有那些作为其子节点的文件。 没有文件可以跨越多个区域。

最后，每个文件都有称为记录的子节点。 和以前一样，该文件恰好包含作为其子节点的记录，并且不能在多个文件中存在记录。 因此，从顶层开始的级别是：

  • 数据库
  • 区
  • 文件
  • 记录

考虑上面给出的示例图，可以单独锁定树中的每个节点。 与2阶段锁定协议一样 ，它应使用共享和独占锁定模式。 当事务以共享或独占模式锁定节点时，事务还会以相同的锁定模式隐式锁定该节点的所有后代。 例如，如果事务T i在独占模式下在文件F c上获得显式锁定，则它在属于该文件的所有记录上具有独占模式的隐式锁定。 它不需要显式锁定F c的各个记录。 这是基于树的锁定和多粒度的分层锁定之间的主要区别。

现在，通过简化锁定文件和记录，系统如何确定是否可以锁定根节点？ 一种可能性是它搜索整个树，但解决方案使多粒度锁定方案的整个目的无效。 获得这种知识的一种更有效的方法是引入一种新的锁定模式，称为意图锁定模式 。

意图模式锁定 -
除了S和X锁定模式外，还有三种具有多种粒度的附加锁定模式：

  • 意图共享（IS）：在树的较低级别显式锁定，但仅限于共享锁。
  • Intention-Exclusive（IX）：使用独占锁或共享锁在较低级别显式锁定。
  • 共享和意向独占（SIX）：以该节点为根的子树在共享模式下显式锁定，显式锁定在较低级别使用独占模式锁进行。

多粒度锁定协议使用意图锁定模式来确保可串行化。 它要求尝试锁定节点的事务T i必须遵循以下协议：

  1. 事务T i必须遵循锁兼容性矩阵。

  2. 事务T 我必须首先锁定树的根，它可以在任何模式下锁定它。

  3. 仅当T i当前将节点的父节点锁定在IX或IS模式时，事务T i才能将节点锁定在S或IS模式。

  4. 仅当T i当前具有以IX或SIX模式锁定的节点的父节点时，事务T i才能将节点锁定在X，SIX或IX模式中。

  5. 事务T i只有在T i先前没有解锁任何节点时才能锁定节点（即，T i是两阶段）。

  6. 事务T i只有在T i当前没有锁定节点的子节点时才能解锁节点。

观察到多粒度协议要求以自上而下（从根到叶）顺序获取锁，而锁必须以自下而上（叶到根）顺序释放。
作为协议的说明，请考虑上面给出的树和事务：

  • 假设事务T 1读取文件F a中的记录R a2 。 然后，T 2需要以IS模式（并按此顺序）锁定数据库，区域A 1和F a ，最后以S模式锁定R a2 。

  • 假设事务T 2修改文件F a中的记录R a9 。 然后，T 2需要在IX模式下锁定数据库，区域A 1和文件F a （并按此顺序），最后在X模式下锁定R a9 。

  • 假设事务T 3读取文件F a中的所有记录。 然后，T 3需要在IS模式下锁定数据库和区域A 1 （并按此顺序），最后在S模​​式下锁定F a 。

  • 假设事务T 4读取整个数据库。 它可以在S模式下锁定数据库后执行此操作。

注意，事务T 1 ，T 3和T 4 可以同时访问数据库 。 事务T 2可以与T 1同时执行，但不能与T 3或T 4同时执行 。
此协议增强了并发性并减少了锁定开销 。在多粒度协议中仍然可以使用解锁，就像在两阶段锁定协议中一样。 这些可以通过使用某些死锁消除技术来消除 。
